# FRE-5 Symphony Postmortem Design

## 背景

`FRE-5` 是 FreshQuant 正式接入 Symphony 后的一条真实开发 issue。它最终产出了已合并的 `PR #65`，但在实现阶段发生了多次阻塞，暴露出 Windows 宿主机服务模式下的运行面问题，而不是单纯的业务代码问题。

当前需要把这次真实运行的经验沉淀成一份可复用文档，供后续真实 issue 与运维排障复用。

## 目标

- 用一份中文文档收口 `FRE-5` 的阻塞原因、恢复过程和最终成功条件
- 明确哪些问题属于运行面，哪些经验值得固化为默认做法
- 给出可执行的配置优化与流程优化建议

## 非目标

- 不修改 Symphony 运行时代码
- 不修改 Linear 状态机
- 不在这一步新增新的 RFC
- 不重复描述 `PR #65` 的业务修复细节

## 候选落点

### 方案 A：写入 `docs/agent/` 独立运行复盘文档（推荐）

优点：

- 最符合“给后续 agent / 新接手开发者复用”的定位
- 能与现有 Symphony 正式治理、宿主机服务部署说明并列
- 便于在索引中直接挂接

缺点：

- 会新增一篇长期维护文档

### 方案 B：只把结论追加到现有治理说明

优点：

- 文件数更少

缺点：

- 治理规则与真实运行复盘会混在一起
- 后续查找具体案例不方便

### 方案 C：只写到 `docs/plans/`

优点：

- 改动最少

缺点：

- `plans` 更偏一次性过程文档，不适合作为长期运行指南入口

## 选型

采用方案 A。

新增一篇 `docs/agent/Symphony真实Issue阻塞复盘与优化建议.md`，内容聚焦：

- 真实时间线
- 阻塞问题分层
- 最终成功经验
- 推荐固化的配置与流程

并同步更新 `docs/agent/index.md`。

## 验收标准

- 文档能解释为什么 `FRE-5` 多次阻塞，但最终仍成功完成
- 文档明确区分运行面问题与业务代码问题
- 文档给出可直接执行的优化建议，而不是抽象感想
- 新文档已接入 `docs/agent/index.md`
