# Daily Screening 筛选工作台优化设计

## 背景

当前 `/daily-screening` 页面已经完成“Dagster 预计算 + 前端自由交集查询”的一轮重构，但仍有四个明显问题：

- `Shouban30 缠论指标` 仍沿用 `/gantt/shouban30` 旧的 `30m` 口径，和用户当前想要的“日线缠论涨幅”不一致。
- `CLS` 条件仍以 12 个模型原始键展示，页面业务语义弱，难以直接对应实际看盘习惯。
- 说明提示分散在每个按钮旁边，重复度高，页面噪音大。
- 查询仍依赖显式点击“查询结果”，交互节奏慢；同时右侧详情区过窄，信息阅读体验差。

这次调整的目标是把 `daily-screening` 收敛成更接近实际选股工作流的查询工作台，并同步修正 `/gantt/shouban30` 上与缠论涨幅周期相关的提示语义。

## 目标

- 把 `daily-screening` 中的 `Shouban30 缠论指标` 改为 `日线缠论涨幅`，真实取数周期切换到 `1d`。
- 把 `CLS` 12 个模型改造成 5 个中文分组展示，组内取并集，组间继续和其他条件取交集。
- 把说明提示从“按钮级”收敛为“分组表头级”悬浮说明。
- 把查询改成自动触发，避免依赖单独的“查询结果”按钮。
- 调整页面宽度分配，让右侧详情区和中间列平分剩余空间。
- 同步更新 `/gantt/shouban30` 页面对缠论涨幅周期的提示，避免两个页面对同一指标的解释冲突。

## 非目标

- 不改 Dagster DAG 拆分结构。
- 不改 `daily_screening_memberships` 的正式存储模型。
- 不把 `CLS` 分组下沉成后端新的 `condition_key` 类型。
- 不改 `daily-screening` 的共享工作区接口语义。

## 已确认口径

### 日线缠论涨幅

- 页面名称从 `Shouban30 缠论指标` 改为 `日线缠论涨幅`。
- 真实计算周期从 `30m` 改为 `1d`。
- 页面展示仍使用 3 个数值字段：
  - `higher_multiple`
  - `segment_multiple`
  - `bi_gain_percent`
- 默认参数展示为：
  - 高级段倍数 `3`
  - 段倍数 `2`
  - 笔涨幅 `%20`
- 这 3 个值不是三个独立条件，而是一个总的筛选组。
- 页面首次打开时，该总按钮默认不选中。
- 只有用户显式选中该组后，这 3 个阈值才一起参与过滤。
- 修改阈值输入框后自动触发查询，但要加短防抖，避免频繁请求。

### CLS 分组与中文名

`CLS` 页面展示改成 5 个业务分组，组内取并集：

- 分组一：二买
  - `01` 类2买
  - `02` 类2买分型
  - `03` 复杂类2买
  - `05` 2买及类2买
- 分组二：三买
  - `04` 3买或中枢3买
- 分组三：压力支撑
  - `06` 低点反弹
  - `07` 顶底互换
- 分组四：背驰
  - `08` 盘整或趋势背驰
  - `09` 下盘下
- 分组五：突破回调
  - `10` 突破回调
  - `11` 突破回踩
  - `12` V反

页面中所有 `CLS` 展示位都改用上述中文名，不再直接面向用户显示 `S0001` 之类的内部模型标识。真实查询时，前端再把分组展开为对应的原始 `cls:*` 条件键。

### 提示与说明

- 不再在每个条件按钮旁单独放 `?` 说明。
- 改为每个筛选分组表头提供悬浮说明。
- 悬浮说明至少包含：
  - 上游数据来源
  - 筛选规则
  - 作用范围
- 顶部总说明里要把“基础池”解释清楚，明确写成：
  - 由 `CLS 各模型结果` 和 `热门 30/45/60/90 天结果` 先取并集形成
  - 上游股票范围为全市场，排除 `ST` 和北交所
  - 用户后续勾选条件时，对当前结果继续取交集

### 自动查询

- `daily-screening` 页面移除“查询结果”按钮。
- 页面首次加载且已有默认 scope 时，自动查询默认结果。
- 点击任意条件后立即查询。
- 修改日线缠论涨幅阈值后自动查询，并加入短防抖。

### 宽度与列表动作

- 左侧筛选区继续保持固定宽度并独立滚动。
- 中间列与右侧详情区平分剩余宽度。
- 中间列内部继续保留“交集列表 + 工作区”上下堆叠。
- `交集列表` 中的 `全部加入 pre_pools` 按钮移到列表头左侧。

## 设计方案

### 1. 后端数据口径

后端真实语义只改一处：`Shouban30` 缠论涨幅的默认周期由 `30m` 改成 `1d`。

当前 `daily-screening` 的 `build_shouban30_chanlun_metrics()` 直接调用 `freshquant.data.gantt_readmodel._resolve_shouban30_chanlun_result()`。因此改动应落在 `gantt_readmodel` 的共享常量和共享解析函数上，而不是只改页面文案。这样可以保证：

- `/daily-screening` 详情与筛选使用的数值是 `1d` 口径。
- `/gantt/shouban30` 页面展示与说明也能同步到同一套真实口径。
- Dagster 每日落库的快照也随之切换到 `1d` 语义，不会出现页面写 `1d`、落库仍是 `30m` 的事实漂移。

字段名保持不变，避免扩大 API 和 Mongo 结构变更面。

### 2. 前端筛选语义

`CLS` 仍由后端返回 12 个原始条件计数，前端负责做“分组映射 + 条件展开”。理由是：

- 正式 membership 真值仍应保持模型粒度，便于排障和详情回溯。
- 五分组只是页面工作台的组织语义，不值得下沉为新的存储结构。
- 可以最小化后端接口变化，把主要复杂度放在前端状态编排层。

`日线缠论涨幅` 则采用“总按钮 + 三个参数”的组合模型：

- 选中前不带任何 metric filter
- 选中后一次性写入三项 metric filter
- 阈值变化时，重新生成同一组 metric filter

### 3. 页面结构

`/daily-screening` 最终保持三栏，但重分配剩余宽度：

- 左：筛选工作台
- 中：交集列表 + 工作区
- 右：标的详情

中、右两列等宽。交集列表头部调整为：

- 左侧：`全部加入 pre_pools`
- 中部：标题与说明
- 右侧：结果条数

### 4. `/gantt/shouban30` 同步调整

`/gantt/shouban30` 不要求这次大改交互，但必须同步更新与缠论涨幅相关的说明文案，明确：

- 当前默认周期是 `1d`
- 高级段倍数、段倍数、笔涨幅都基于 `1d` 结构计算

否则会出现同一套指标在两个页面中名称一致、周期理解不一致的问题。

### 前端状态层

- 锁定组内取并集的查询展开逻辑
- 锁定 12 个模型中文名映射
- 锁定“日线缠论涨幅”总按钮开关与默认阈值语义

### 文档

## 风险与回避

- 风险一：`/gantt/shouban30` 与 `daily-screening` 提示不同步
  - 回避：把周期切换放在共享后端口径上，并同步更新两个页面文案
- 风险二：自动查询导致频繁请求
  - 回避：数值输入增加短防抖；普通按钮点击保留即时查询
- 风险三：`CLS` 分组仅前端映射，详情展示出现前后不一致
  - 回避：统一通过同一个中文映射函数渲染筛选区和详情区

## 实施边界

这次只做：

- `1d` 缠论涨幅口径切换
- `CLS` 分组与中文名展示
- 分组级说明提示
- 自动查询
- 页面宽度与动作布局调整
- `/gantt/shouban30` 提示同步

这次不做：

- 新增后端条件模型类型
- 新增页面执行入口
- 调整 Dagster 节点结构
