# 前端高密工作台与导航标签设计

## 背景

当前 `morningglory/fqwebui` 已经有多页工作台页面，但布局策略并不统一：

- `MyHeader.vue` 仍使用当前页 `router.replace`，不能打开浏览器新标签页，也没有统一的页面标题真值。
- `StockControl.vue` 仍保留“持仓股列表”，不符合当前双监控模式需求。
- `GanttShouban30Phase1.vue`、`RuntimeObservability.vue`、`SystemSettings.vue` 以及若干 workbench 页面存在 `100vh / min-height: 100vh` 与内容自然撑高并存的情况，容易把工作区或详情区挤出屏幕，形成整页滚动。
- `runtime-observability` 与 `system-settings` 在部分区域使用卡片化承载，信息密度不足。

本次目标是在桌面 `3440x1440` 分辨率下，把关键前端页面收口为“一屏工作台 + 区域内滚动”的稳定形态，同时把导航行为改成“新标签页 + 正确标题”。

## 目标

- 导航栏按钮打开浏览器新标签页，并在浏览器标签上展示对应按钮文字。
- `stock-control` 调整为左右两种监控模式：左侧为“持仓股信号 + must_pools买入信号”，右侧为“stock_pools模型信号”。
- `gantt/shouban30` 在一屏内固定展示“首板板块 / 热点标的 / 标的详情 / 工作区”，滚动发生在列表内而不是页面。
- `runtime-observability` 中间“全局 Trace”改为高密度列表承载，保留现有信息，不遗漏字段。
- `system-settings` 摘要区改为列表型承载，提升信息密度。
- 修正 `gantt/shouban30` 中“韭研公式”为“韭研公社”。
- 对其他主要前端页面做一轮公共布局检查，避免继续出现屏幕级滚动条。

## 非目标

- 不重做整套前端视觉语言。
- 不引入新的全局状态管理方案。
- 不改变现有 API 协议和数据结构。
- 不为移动端强行实现“绝对一屏无滚动”；移动端保留单列堆叠和局部滚动。

## 方案选择

### 方案 A：逐页打补丁

只在每个页面里补高度、滚动和按钮逻辑。

优点：

- 改动局部，起步快。

缺点：

- 布局策略继续分散。
- 后续新增页面仍容易重新长出屏幕滚动。
- 标题与导航映射没有统一真值。

### 方案 B：轻量统一壳子 + 页面定点改造（采用）

保留现有页面数据逻辑，统一收口三类公共能力：

- 导航映射与页面标题真值
- 视口固定、内容区内滚动的页面壳子
- 列表/摘要的高密承载样式

然后针对 `stock-control`、`gantt/shouban30`、`runtime-observability`、`system-settings` 做结构调整。

优点：

- 能同时解决当前需求和一部分共性布局问题。
- 风险比整站重构低。
- 便于后续页面沿用统一模式。

缺点：

- 会碰到若干公共文件，需要回归多个页面。

### 方案 C：完整重构全局工作台框架

重做全局 shell、导航、标签管理与页面容器。

优点：

- 长期最整齐。

缺点：

- 本次需求来说过重，回归面过大。

## 采用方案

采用方案 B。

## 设计

### 1. 导航与标题真值

新增一份页面元数据映射，统一维护：

- 路由路径
- 导航按钮文案
- 页面标题

`MyHeader.vue` 不再调用 `router.replace`，改为：

- 通过 `router.resolve` 生成目标 URL
- 附带 `tabTitle` query
- 使用 `window.open(url, '_blank', 'noopener')` 打开新标签页

页面标题通过路由统一同步：

- 优先使用 `route.query.tabTitle`
- 否则回退到 route meta title
- 如果都没有则使用默认站点标题

这样可保证：

- 从导航栏打开的新标签页标题与按钮文案一致
- 直接访问路由时也能有稳定标题

### 2. 统一的一屏工作台布局

公共布局策略改为：

- 页面根容器使用 `height: 100dvh`
- 页面根容器 `overflow: hidden`
- Header 固定占顶
- 页面 body 使用 `flex` 或 `grid` 吃满剩余高度
- 真正需要滚动的区域只放在列表、表格、详情区内

对已有 `workbench-density.css` 做增强，让 `workbench-page` 类页面默认具备上述行为；对未使用该样式的页面在各自样式里补齐同样约束。

桌面目标分辨率优先按 `3440x1440`，移动端在 media query 下回退为单列和局部滚动。

### 3. Stock Control 页面

页面改成两列：

- 左列：上“持仓股信号”，下“must_pools买入信号”
- 右列：“stock_pools模型信号”

删除 `StockPositionList` 在该页的入口，不再显示“持仓股列表”。

两列都要是独立容器：

- 顶部标题固定
- 列表区滚动
- 页面本身不滚动

这样左/右分别承载“持仓/买入监控模式”和“模型监控模式”。

### 4. Gantt / Shouban30 页面

页面改成“上三下一区”的固定工作台：

- 上排：首板板块 / 热点标的 / 标的详情
- 下排：工作区

关键变化：

- `工作区` 固定保留在视口内，不再因为前三块过高被挤到屏幕外
- 四块都使用 `min-height: 0` + 区域内滚动
- `panel-table`、workspace tabs、详情表格都只在自身滚动
- 相关 provider 文案统一使用“韭研公社”

### 5. Runtime Observability 页面

保持三栏：

- 左：组件导航
- 中：全局 Trace / 组件 Event
- 右：详情

中间栏改造重点：

- “全局 Trace”从卡片式堆叠改为高密度列表
- 保留现有的状态、标题、副标题、身份标识、时间、trace kind、steps、duration、路径、guardian tags、结论、flow nodes 等信息
- 通过更紧凑的行结构与列式信息承载提升密度
- 列表区独立滚动，不滚整个页面

右侧详情不改信息内容，只调整容器高度与滚动关系。

### 6. System Settings 页面

保留编辑表单，但把右侧摘要/字典区域从卡片堆改为列表型区域：

- 启动配置摘要改为分组列表
- 系统设置摘要改为分组列表
- 策略字典改为高密度列表

页面整体也改成固定视口工作台：

- 顶部说明区收紧
- 主体区域固定高度
- 表单区与摘要区分别局部滚动

## 测试策略

- 为导航标题映射提取纯函数并添加 node test，验证按钮文案、路由、标题同步逻辑。
- 为 `stock-control` 新增/更新结构测试，验证持仓股列表已移除、左右监控结构存在。
- 为 `runtime-observability` 新增纯函数或结构测试，验证全局 Trace 列表行仍保留关键信息字段。
- 为 `system-settings` 新增结构测试，验证摘要区使用列表承载而不是卡片类名。
- 为 `gantt/shouban30` 新增结构测试，验证工作区固定区域与 provider 文案。
- 跑前端构建，确保样式与动态导入未破坏。

## 风险与回避

- 风险：公共布局样式调整可能影响订单、仓位、TPSL 等页面。
  回避：优先修改通用容器行为，随后回归主要 workbench 页面构建与关键结构。

- 风险：`window.open` 在部分调用路径被浏览器拦截。
  回避：只从用户点击事件同步触发打开。

- 风险：`RuntimeObservability` 模板较大，局部改造容易漏字段。
  回避：先补测试，再重构模板中间栏。

- 风险：`SystemSettings` 当前视觉样式偏卡片化，改高密列表后可读性下降。
  回避：保留分组标题与轻量分隔，不保留大块卡片容器。
