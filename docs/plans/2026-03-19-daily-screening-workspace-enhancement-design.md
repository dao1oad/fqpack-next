# 每日选股工作区增强设计

## 背景

当前每日选股页面已经切换为“Dagster 预计算 + 前端交集查询”的工作台，但和 `/gantt/shouban30` 相比，仍然缺少两类关键能力：

- 交集结果无法继续沉淀到共享工作区
- 页面缺少足够明确的条件提示，用户无法直接从界面理解上游来源、筛选规则、并集和交集语义

这导致每日选股页面虽然能查结果，但无法承接后续人工操作，也不够自解释。

## 目标

- 在每日选股页面增加与 `/gantt/shouban30` 共用的工作区能力
- 所有工作区数据复用 `/gantt/shouban30` 同一套 `pre_pools / stock_pools / must_pools`
- 交集列表支持“全部加入 pre_pools”
- 在筛选工作台中补充总说明和每个条件的点击式提示说明

## 非目标

- 不为每日选股单独新建一套工作区存储
- 不新增一套 daily-screening 专属的工作区后端接口语义
- 不把条件说明直接铺成长文常驻在页面上

## 已确认口径

- 工作区操作严格作用于“当前交集结果”
- 用户未选择任何条件时，“当前交集结果”就是当前基础池结果
- 每日选股页面与 `/gantt/shouban30` 共用同一套工作区数据
- 工作区批量操作和行级操作都复用 `/gantt/shouban30` 原样逻辑
- 每个筛选条件旁边增加可点击提示图标
- 页面提示文案不使用 `A ∪ B` 这类代号表达

## 方案比较

### 方案 1：前端直接复用 `/gantt/shouban30` 工作区接口与构造逻辑

- 每日选股页面直接调用 `ganttShouban30.js`
- 直接复用 `shouban30PoolWorkspace.mjs`
- 交集列表结果转成 `appendShouban30PrePool` 所需 payload

优点：

- 复用最多
- 与 `/gantt/shouban30` 行为完全一致
- 后端改动最小

缺点：

- 每日选股页面会直接依赖 gantt 工作区接口

### 方案 2：daily-screening 后端新增一层工作区代理接口

- 页面只调 `/api/daily-screening/**`
- 后端再转调共享工作区服务

优点：

- 页面 API 看起来更统一

缺点：

- 只是多包一层
- 容易复制已有接口语义和测试
- 实际收益很小

## 结论

采用方案 1。

原因很直接：你已经明确要求共用同一套池子数据，并且复用 `/gantt/shouban30` 原样逻辑。既然如此，就不值得再加一层 daily-screening 专属代理接口。

## 页面设计

### 1. 筛选工作台说明

在筛选工作台顶部补一段总说明，明确：

- 上游股票范围是全市场，排除 `ST` 和北交所
- 基础池由 `CLS 各模型结果` 和 `热门 30/45/60/90 天结果` 先取并集形成
- 用户后续勾选的条件，都是在当前结果上继续取交集
- 交集结果可以继续沉淀到 `pre_pools -> stock_pools -> must_pools`

### 2. 条件级提示

每个筛选条件旁边放一个可点击提示图标。

点击后显示统一结构：

- 上游数据来源
- 筛选规则
- 结果作用范围

说明结构统一，但每个条件的文案不同。

### 3. 交集列表增强

交集列表头部新增按钮：

- `全部加入 pre_pools`

按钮作用范围固定为当前交集结果：

- 无条件时加入当前基础池结果
- 有条件时加入当前交集结果

### 4. 工作区区块

在交集列表下方新增 `工作区` 面板，包含两个标签：

- `pre_pools`
- `stock_pools`

行为对齐 `/gantt/shouban30`：

- `pre_pools`
  - 同步到 `stock_pools`
  - 同步到通达信
  - 清空
  - 行级：加入 `stock_pools` / 删除
- `stock_pools`
  - 同步到 `must_pools`
  - 同步到通达信
  - 清空
  - 行级：加入 `must_pools` / 删除

## 数据与交互设计

### 1. 交集结果转工作区 payload

每日选股交集列表需要补一个适配层，把当前结果行转成共享工作区 `append pre_pool` 所需结构：

- `code6`
- `name`
- `category`
- `remark`
- 必要的来源说明

这里不复用 `/gantt/shouban30` 的“按板块收集股票” payload 构造，而是针对每日选股结果单独做一层“按当前结果列表批量构造”的 helper。

### 2. 工作区数据来源

工作区列表直接读取共享接口：

- `getShouban30PrePool`
- `getShouban30StockPool`

所有批量与行级动作也直接使用共享接口。

### 3. 条件提示元数据

每日选股条件目录在前端规范化时，补充每个条件的提示元数据：

- `source`
- `rule`
- `scope_note`

页面层只负责渲染，不再在模板里硬编码一堆条件文案。

## 测试策略

### 1. 前端纯逻辑测试

在 `dailyScreeningPage.test.mjs` 覆盖：

- 条件目录能规范化出提示信息
- 当前交集结果能构造出追加到 `pre_pools` 的 payload
- 工作区 tabs 能复用共享工作区结构

### 2. 页面交互测试

覆盖：

- 页面渲染总说明
- 每个条件渲染提示入口
- “全部加入 pre_pools” 只使用当前结果集
- 工作区区域显示 `pre_pools / stock_pools`

### 3. 回归测试

确保不破坏现有：

- scope 加载
- 条件查询
- 详情展示

## 部署影响

- `morningglory/fqwebui/**` 改动后，需重建 Web UI
- 如果只复用现有共享工作区接口，后端无需新增部署面
- 部署后需人工确认每日选股页面和 `/gantt/shouban30` 页面看到的是同一套工作区数据
