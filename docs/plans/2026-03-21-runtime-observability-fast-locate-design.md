# Runtime Observability 异常快速定位设计

## 背景

`/runtime-observability` 当前已经能展示：

- 时间范围内的全局异常链路数
- 时间范围内的异常节点总数
- 各组件的异常链路数/异常节点数
- Trace 详情中的步骤列表

但这些数字现在只是摘要展示。用户看到例如：

- `异常链路 10`
- `异常节点 390`
- `guardian_strategy` 卡片中的 `异常 8/16`

仍然需要手工切视图、手工挑 Trace、再手工找异常节点，排障路径过长。

## 目标

让用户从“异常数字”在 1 到 3 次操作内直接落到对应异常集合、对应组件、对应节点。

## 非目标

- 不新增后端接口
- 不改 ClickHouse 聚合口径
- 不重做整页布局
- 不改变现有时间范围与高级筛选的查询语义

## 选型结论

采用“数字即入口”的最小改动方案。

### 方案 A：数字直接跳转到异常集合

这是本次采用的方案。

- 顶部 `异常链路` 改成可点击入口
- 顶部 `异常节点` 改成可点击入口
- 组件卡片中的异常摘要拆成可点击入口
- Trace 步骤区增加异常导航按钮

优点：

- 复用现有三栏布局和现有 API
- 用户操作路径最短
- 改动集中在前端，风险低

### 方案 B：新增异常导航面板

不采用。

原因：

- 会引入新的中间态面板
- 页面复杂度上升
- 对当前需求来说收益不够高

### 方案 C：顶部数字弹出 Drawer/Popover

不采用。

原因：

- 比直接跳转多一层操作
- 排障路径仍然偏长

## 交互设计

### 1. 顶部摘要区

#### `异常链路 N`

点击后：

- 切换到 `全局 Trace`
- 自动打开 `仅异常`
- 清空 trace kind 局部选择，避免被旧 kind 卡住
- 保留当前时间范围与高级筛选条件
- 自动选中第一条异常 Trace
- 右侧详情默认落在 `步骤` tab
- 默认选中首个异常节点

#### `异常节点 N`

点击后：

- 切换到 `全局 Trace`
- 自动打开 `仅异常`
- 自动选中第一条异常 Trace
- 右侧详情直接落在 `步骤` tab
- 自动定位首个异常节点

### 2. 组件卡片

把当前 `异常 8/16` 变为两个明确入口：

- `异常链路 8`
- `异常节点 16`

点击行为：

#### `异常链路 X`

- 切到 `全局 Trace`
- 自动打开 `仅异常`
- 只保留“该组件发生过异常”的 Trace
- 自动选中该组件对应的第一条异常 Trace
- 右侧默认定位首个异常节点

#### `异常节点 Y`

- 切到 `组件 Event`
- 自动打开 `仅异常`
- 自动选中该组件的首条异常 Event

### 3. Trace 步骤区异常导航

在步骤区工具栏增加：

- `首个异常`
- `上一个异常`
- `下一个异常`
- `最慢节点`

行为：

- `首个异常`：直接选中第一条异常步骤
- `上一个异常`：相对当前选中步骤向前跳
- `下一个异常`：相对当前选中步骤向后跳
- `最慢节点`：跳到最长耗时步骤

### 4. 自动滚动

当通过上述入口完成定位时：

- 步骤列表应自动滚动到当前选中步骤
- 保证用户不需要再手动在长表内查找

## 状态设计

新增一个仅用于 Trace 视图的本地聚焦状态：

- `traceIssueFocus.component`

用途：

- 支撑“按组件异常链路聚焦”
- 不污染高级筛选 `query`
- 不影响 Event 视图已有的 `boardFilter`

现有状态继续复用：

- `onlyIssues`
- `activeView`
- `selectedTrace`
- `selectedStep`
- `selectedEvent`
- `selectedTraceKind`

## 数据与实现边界

本次不新增 API，完全复用：

- `/api/runtime/traces`
- `/api/runtime/events`
- `/api/runtime/traces/<trace_key>`

前端新增的逻辑主要是：

- 对已加载 Trace 做“异常组件聚焦”过滤
- 在详情步骤中做异常步导航
- 把摘要数字改成操作入口

### 视图结构测试

## 风险

- 当前 Trace 列表是分页加载的，组件异常链路聚焦只作用于当前已加载结果集
- 若时间范围极大、异常非常多，用户可能仍需继续点“加载更多 Trace”

这属于当前页面已有分页边界，不是本次交互改动新增的问题。
