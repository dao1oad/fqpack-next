# 实盘对接说明

> 本文描述目标仓 `D:\fqpack\freshquant-2026.2.23` 当前实盘运行面，已不再适用旧仓“直接写队列 + `stock_fills` 为主事实”的口径。

## 1. 当前实盘链路

当前执行面仍以 MiniQMT / XTQuant 为核心，但中间层已经拆成独立域：

- 订单管理：`freshquant/order_management/`
- 仓位管理：`freshquant/position_management/`
- 止盈止损：`freshquant/tpsl/`
- XTData 实时：`freshquant/market_data/xtdata/`

主链路：

```text
Guardian / API / CLI
  -> OrderSubmitService
  -> Redis 订单队列
  -> 宿主机 broker
  -> MiniQMT / XTQuant
  -> XT 回报 ingest / reconciliation
  -> 订单主账本 + 兼容投影
```

## 2. 主要组件

### 2.1 策略与上游入口

常见入口：

```powershell
python -m freshquant.signal.astock.job.monitor_stock_zh_a_min
```

- `POST /api/order/submit`
- `POST /api/order/cancel`
- `POST /api/stock_order`（兼容入口）
- `fqctl om-order submit`
- `fqctl om-order cancel`

### 2.2 宿主机 broker

broker 必须运行在安装 MiniQMT / XTQuant 的 Windows 宿主机：

```powershell
python -m fqxtrade.xtquant.broker
```

职责：

- 监听 Redis 订单队列
- 调用 XT / MiniQMT 下单和撤单
- 接收委托、成交、资产、持仓回报
- 将 XT 回报写回订单域和兼容集合

### 2.3 仓位管理

策略单提交前会读取融资账户仓位状态：

```powershell
python -m freshquant.position_management.worker
```

当前语义：

- `source=strategy` 的订单会走仓位门禁
- 人工/API/CLI 手工单默认旁路

### 2.4 TPSL

止盈止损已拆成独立模块：

```powershell
python -m freshquant.tpsl.tick_listener
```

它不再沿用旧 `grid_configs / stoploss_configs / fill_stoploss_configs` 执行链。

## 3. XTData 实时链路

宿主机实时链路：

```powershell
python -m freshquant.market_data.xtdata.market_producer
python -m freshquant.market_data.xtdata.strategy_consumer --prewarm --max-bars 20000
```

当前语义：

- Producer 只订阅 `load_monitor_codes(...)`
- 不再额外并入 TPSL 活跃标的池
- `stock_realtime` 与 `index_realtime` 统一只存 `raw/bfq`
- qfq 不再在写 realtime 时完成，而是读取窗口时统一做一次

### 3.1 realtime 集合语义

- `freshquant.stock_realtime`
  - 新语义：股票 realtime 统一只存 raw/bfq
- `freshquant.index_realtime`
  - 保持 raw/bfq，不再和股票链路分叉

读取链路统一为：

```text
history raw/bfq
  + realtime raw/bfq
  -> merge / dedupe / sort
  -> apply base stock_adj / etf_adj
  -> apply intraday override (if exists)
```

注意：

- 任何直接读取 `stock_realtime` 并假设“股票 realtime 已经是 qfq”的脚本都需要调整
- 切换到该版本时，必须清理或重建当天 `stock_realtime / index_realtime`

## 4. 参考数据任务（reference-data）

宿主机当前有两类盘前参考数据任务，统一由 supervisor `group:fqnext_reference_data` 管理，但仍保持独立进程：

### 4.1 融资标的同步

```powershell
python -m freshquant.order_management.credit_subjects.worker
python -m freshquant.order_management.credit_subjects.worker --once
```

职责：

- 通过 XT 信用连接同步融资标的
- 写入 `freshquant_order_management.om_credit_subjects`
- 启动后先跑一次，常驻时默认每天 `09:20` 再跑一次

### 4.2 盘前除权覆盖刷新

```powershell
python -m freshquant.market_data.xtdata.adj_refresh_worker
python -m freshquant.market_data.xtdata.adj_refresh_worker --once
```

职责：

- 读取 `load_monitor_codes(...)`
- 保持 TDX/pytdx 盘后 `stock_adj / etf_adj` 为 authoritative source
- 盘前通过 XTData `front/raw` 对照计算当日 `intraday override`
- 写入：
  - `quantaxis.stock_adj_intraday`
  - `quantaxis.etf_adj_intraday`
- 启动后先跑一次，常驻时默认每天 `09:20` 再跑一次

`intraday override` 只负责“当前交易日锚点修正”，不直接写 qfq K 线。

## 5. 当前配置来源

FreshQuant 当前不是只靠 Mongo `params`：

- Dynaconf / 仓库根 `.env`
- Mongo `freshquant.params`
- 独立分库配置：
  - `order_management.*`
  - `position_management.*`

宿主机 broker / XTData 若与 Docker 并行环境联动，必须显式对齐：

```text
FRESHQUANT_MONGODB__HOST=127.0.0.1
FRESHQUANT_MONGODB__PORT=27027
FRESHQUANT_REDIS__HOST=127.0.0.1
FRESHQUANT_REDIS__PORT=6380
```

否则会回落到默认 `27017/6379`，导致宿主机与 Docker 读写两套 Mongo/Redis。

## 6. 数据事实口径

需要特别注意：

- `stock_fills` 已降级为兼容投影
- 当前订单事实优先看：
  - `internal_order_id`
  - `request_id`
  - `buy_lot_id`
  - `trade_fact_id`
- 当前持仓读取优先走订单域投影，再按需看 `xt_positions`

排障时不要再默认 `freshquant.stock_fills` 是唯一主事实表。

## 7. Docker 并行模式补充

当本仓使用 `docker/compose.parallel.yaml` 并行部署时：

- Docker 内 API / Dagster 读取 `fq_mongodb`
- 宿主机映射端口：
  - Mongo：`27027`
  - Redis：`6380`
- `broker`、`xtdata producer`、`xtdata consumer`、`credit_subjects.worker`、`adj_refresh_worker` 仍运行在 Windows 宿主机

首次切换到 Docker Mongo 时，先执行：

```powershell
$env:FRESHQUANT_MONGODB__HOST="127.0.0.1"
$env:FRESHQUANT_MONGODB__PORT="27027"
python -m freshquant.initialize --quiet
```

如果宿主机旧库中已经存在 `freshquant.params`，应先同步到 Docker 的 `freshquant.params`，再重启宿主机进程。
