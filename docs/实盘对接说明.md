# 实盘对接说明

> 本文描述的是目标仓 `D:\fqpack\freshquant-2026.2.23` 当前实盘链路，不再适用旧仓“直接写队列 + `stock_fills` 为主事实”的口径。

## 1. 当前实盘链路概览

当前实盘对接仍以 **MiniQMT / XTQuant** 为执行面，但目标仓已经在中间层补上了：

- 订单域受理层 `freshquant/order_management/`
- 仓位管理 `freshquant/position_management/`
- 独立止盈止损 `freshquant/tpsl/`

当前主链路可以概括为：

```text
Guardian / API / CLI
  -> OrderSubmitService
  -> Redis 订单队列
  -> 宿主机 broker
  -> MiniQMT / XTQuant
  -> XT 回报 ingest / reconciliation
  -> 订单主账本 + 兼容投影
```

## 2. 组成组件

### 2.1 策略与上游入口

常见上游入口：

1. Guardian

```powershell
python -m freshquant.signal.astock.job.monitor_stock_zh_a_min
```

2. HTTP API

- `POST /api/order/submit`
- `POST /api/order/cancel`
- `POST /api/stock_order`（兼容入口）

3. CLI

- `fqctl om-order submit`
- `fqctl om-order cancel`
- `xtquant buy/sell/cancel`（兼容桥接）

### 2.2 订单受理层

统一入口位于：

- `freshquant/order_management/submit/service.py`

核心职责：

- 规范化 `action / symbol / price / quantity`
- 为请求生成 `request_id / internal_order_id`
- 对 `source=strategy` 的订单先走仓位管理门禁
- 将最终执行 payload 推入 Redis 订单队列

### 2.3 宿主机 broker

broker 必须运行在 **MiniQMT 同机的 Windows 宿主机** 上：

```powershell
python -m fqxtrade.xtquant.broker
```

实际代码路径：

- `morningglory/fqxtrade/fqxtrade/xtquant/broker.py`

职责：

- 监听 Redis 订单队列
- 调用 XT / MiniQMT 下单和撤单
- 接收委托、成交、资产、持仓回报
- 把 XT 回报写回订单域与兼容集合

### 2.4 仓位管理

策略单会在下单前读取融资账户仓位状态：

- 模块：`freshquant/position_management/`
- worker：

```powershell
python -m freshquant.position_management.worker
```

当前语义：

- 仅 `source=strategy` 的订单会走仓位门禁
- 人工/API/CLI 手工单默认旁路

### 2.5 TPSL

止盈止损已拆分为独立模块：

- `freshquant/tpsl/`
- worker：

```powershell
python -m freshquant.tpsl.tick_listener
```

它不再沿用旧仓 `grid_configs / stoploss_configs / fill_stoploss_configs` 的执行链。

## 3. 当前配置来源

FreshQuant 当前不是只靠 Mongo `params`：

- Dynaconf / 仓库根 `.env`
- Mongo `freshquant.params`
- 独立分库配置：
  - `order_management.*`
  - `position_management.*`

宿主机 broker / XTData 运行时若与 Docker 并行环境联动，必须显式对齐：

```text
FRESHQUANT_MONGODB__HOST=127.0.0.1
FRESHQUANT_MONGODB__PORT=27027
FRESHQUANT_REDIS__HOST=127.0.0.1
FRESHQUANT_REDIS__PORT=6380
```

否则会回落到默认 `27017/6379`，导致 Docker 内 API 与宿主机 MiniQMT 链路读写两套 Mongo/Redis 数据。

## 4. 当前数据事实口径

需要特别注意：

- `stock_fills` 已降级为**兼容投影**
- 当前订单事实应优先看：
  - `internal_order_id`
  - `request_id`
  - `buy_lot_id`
  - `trade_fact_id`
- 当前持仓读取优先走订单域投影，再按需补 `xt_positions`

排障时不要再假设 `freshquant.stock_fills` 是唯一主事实表。

## 5. Docker 并行模式补充

当本仓库使用 `docker/compose.parallel.yaml` 并行部署时：

- Docker 内 API / Dagster 读取 `fq_mongodb`
- 宿主机映射端口：
  - Mongo：`27027`
  - Redis：`6380`
- `broker`、`xtdata producer`、`xtdata consumer` 仍运行在 Windows 宿主机

首次切换到 Docker Mongo 时，先执行：

```powershell
$env:FRESHQUANT_MONGODB__HOST="127.0.0.1"
$env:FRESHQUANT_MONGODB__PORT="27027"
python -m freshquant.initialize --quiet
```

如果宿主机旧库中已经存在 `freshquant.params`，应先同步到 Docker 的 `freshquant.params`，再重启宿主机 `broker / producer / consumer`。
