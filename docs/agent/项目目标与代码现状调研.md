# 项目目标与代码现状调研

- **调研日期**：2026-03-05

## 1. 本期项目目标（迁移与重构）

本仓库的明确目标来自 `AGENTS.md` 与 `docs/migration/README.md`：

- 本仓库是 **目标架构**：在“重构与去冗余”的前提下，逐步迁移旧分支仓库 `D:\fqpack\freshquant` 的功能。
- **允许破坏性变更（Breaking Changes）**，但任何新增模块/入口/跨目录重构/新增依赖/破坏性变更，都必须先写 RFC（`docs/rfcs/`）并评审通过后才能编码。
- 迁移进度与破坏性变更采用“文件化治理”：
  - `docs/migration/progress.md`：按 RFC 粒度跟踪进度
  - `docs/migration/breaking-changes.md`：落地破坏性变更必须登记并引用 RFC

> 现状结论：当前 `docs/migration/progress.md` 仍为空表，`docs/rfcs/` 仅有模板与 README，尚未登记任何 RFC（即迁移治理体系已搭好，但迁移单元尚未开始登记）。

## 2. 仓库结构与组成（Monorepo）

本仓库属于“**Monorepo + 本地依赖**”形态：核心业务包 + 多子项目（微服务/算法包/交易对接）+ vendored 第三方源码。

### 2.1 顶层目录速览

（基于 `git ls-files` 的粗粒度统计）

- `freshquant/`：主 Python 包（约 298 个 tracked 文件；约 293 个 `.py`）
- `morningglory/`：子项目集合（约 458 tracked 文件；约 91 个 `.py`）
- `sunflower/`：vendored 第三方依赖源码（约 1167 tracked 文件；约 652 个 `.py`）
- `ikebana/`：Windows 常驻/托盘类工具（用于周期拉起本地任务）
- `script/`：脚本与 CI 校验
- `docker/`：后端与前端镜像构建文件
- `deps/`：二进制依赖（如 TA-Lib 安装包）
- `docs/`：文档（含迁移治理与 agent 文档）

### 2.2 Python 包与安装方式

主包与子项目通过 `install.py` / `docker/Dockerfile.rear` 以“本地路径安装”的方式装入环境，典型顺序：

- 缠论算法包：`morningglory/fqchan01..06/python`（当前包含 01/02/03/04/06）
- 辅助包：`morningglory/fqcopilot/python`
- 工作流：`morningglory/fqdagster`
- 交易对接：`morningglory/fqxtrade`
- vendored 依赖：`sunflower/xtquant`、`sunflower/backtrader`、`sunflower/pytdx`、`sunflower/QUANTAXIS`
- 主包：`freshquant/`

## 3. 关键入口与运行形态（Public Interfaces）

### 3.1 CLI：`fqctl`

- **入口**：`pyproject.toml` → `fqctl = freshquant.cli:main`
- **命令组注册**：`freshquant/cli.py`
  - 数据：`stock.*` / `etf.*` / `index.*` / `bond.*` / `future.*`
  - 实盘：`xt-asset` / `xt-trade` / `xt-order` / `xt-position`
  - 其他：`channel`

> 注意：CI 的治理脚本对 CLI/配置等区域有额外门禁（见 §6）。

### 3.2 HTTP 服务（后端）

- **TDX 行情网关（Tornado）**：`python -m freshquant.gateway.tdxhq --port 5001`
  - 文件：`freshquant/gateway/tdxhq.py`
  - 作用：以 HTTP 封装 `pytdx` 的行情/列表/分钟线/逐笔等接口（含股票与期货扩展行情 `ex/*`）。
- **业务 API Server（Flask + gevent）**：`python -m freshquant.rear.api_server`
  - 文件：`freshquant/rear/api_server.py`
  - 作用：注册 `freshquant.rear.{stock,future,general}.routes` 蓝图，提供业务接口。

### 3.3 数据采集与后台任务

典型“在线运行”组件（见 `deploy_rear.bat` / `deploy.bat`）：

- **行情采集**：`freshquant/market_data/*`（例如 `stock_cn_a_collector.py`、`stock_cn_a_sina_tick_collector.py`）
- **信号监控（Guardian）**：`python -m freshquant.signal.astock.job.monitor_stock_zh_a_min`
  - 读取监控标的 → 调 `freshquant.chanlun_service.get_data_v2` → 生成信号 → 写入 MongoDB/推送队列
- **Huey Worker（Redis 队列）**：`python -m freshquant.worker.consumer freshquant.worker.queue.huey -w 50`
- **Dagster**：`dagster-webserver` / `dagster-daemon run`（配置来源 `morningglory/fqdagsterconfig`）

### 3.4 Web UI（前端）

- `morningglory/fqwebui/`：Vue 3 + Vite + Element Plus + ECharts
- 镜像：`docker/Dockerfile.web`（构建后由 Nginx 承载静态资源）
- 部署脚本：`deploy_web.bat`

### 3.5 Windows 常驻工具（本地对接）

- `ikebana/CommandTrayHost/config.json`：定义定时拉起任务（例如 `fqxtrade.exe xtquant auto`、`ensure_xt_mini_qmt.py`）
- 场景：MiniQMT 必须运行在本机（同一台机器）时，用托盘程序做“守护+拉起”更贴合桌面环境。

## 4. 配置与参数体系（Data/Config）

### 4.1 Dynaconf（文件 + 环境变量）

- 初始化：`freshquant/config.py` → `settings = Dynaconf(...)`
- 配置文件搜索顺序（覆盖优先级由 dynaconf 决定，常用路径见 `docs/agent/配置管理指南.md`）：
  - 包内：`freshquant/freshquant.yaml`（默认配置）
  - EXE_DIR / CWD / `~/.freshquant/` 等位置的 `freshquant.{yaml,yml,json}` 与 `freshquant_*.{yaml,yml,json}`
- 环境变量前缀：`envvar_prefix="freshquant"`（Docker `.env` 当前使用 `FRESHQUANT_...`，Windows 下大小写不敏感，但 Linux 容器内建议统一为大写以避免歧义）

### 4.2 MongoDB Params（运行时参数）

除了 dynaconf，仓库还有“运行时参数”体系：

- 集合：`DBfreshquant["params"]`
- API：`freshquant/carnation/param.py` → `queryParam("a.b.c", default)`
- 示例：`freshquant/gateway/__init__.py` 用 `queryParam('tdx.hq.endpoint', ...)` 读取 TDX 网关地址

> 现状要点：目前同时存在“配置文件/环境变量（dynaconf）”与“数据库参数（params）”两套来源；后续如要统一/分层，需要以 RFC 明确边界（哪些属于部署配置，哪些属于可运营参数）。

## 6. CI 与治理门禁（现状即规则）

GitHub Actions（` .github/workflows/ci.yml`）包含三类检查：

1) `CI / governance`：`python script/ci/check_governance.py ...`
   - 改动触及 `freshquant/`、`morningglory/`、`sunflower/`、`script/` 等“迁移/代码区域”时，**必须同步更新** `docs/migration/progress.md`
   - 改动触及 `freshquant/cli.py`、`freshquant/config.py`、`freshquant/command/`、`freshquant/rear/` 等“可能公共接口区域”时，**必须同步更新** `docs/migration/breaking-changes.md`（即便写“无破坏性变更”也要留痕）
2) `CI / pre-commit`：对 PR diff 运行 `black/mypy/isort/...`
3) `CI / pytest`：仅运行 `python -m pytest -q freshquant/tests`（注意：当前 `freshquant/tests` 用例非常少）

## 7. 当前代码现状观察（用于后续 RFC 的问题池）

- **测试现状**：CI 只跑 `freshquant/tests`，目前仅 1 个用例；存在 `freshquant/test`（legacy）但未纳入 CI。
- **依赖形态**：`pyproject.toml` 依赖列表较大，且同时 vendored 源码（`sunflower/*`）+ 本地路径安装（`morningglory/*`）。
- **多运行框架并存**：Tornado（TDX 网关）+ Flask（rear API）+ Dagster（工作流）+ Huey（worker）+ APScheduler（`freshquant/scheduler.py`）。
- **配置来源分裂**：dynaconf + MongoDB params 并存，且存在大小写环境变量命名差异（Windows vs Linux 容器需特别小心）。
- **重复/待收敛**：如时区、TIME_DELTA 等配置在 `freshquant/config.py` 与 `freshquant/carnation/config.py` 之间存在重复（`carnation/config.py` 已明确提示“之后要统一到这里来”）。

## 7.1 2026-03-07 追加：仓位管理模块现状

- 已新增独立模块 `freshquant/position_management/`，职责边界与订单管理模块一致，单独使用 MongoDB 分库 `freshquant_position_management`。
- 仓位状态的唯一事实口径是融资账户 `query_credit_detail(account).m_dEnableBailBalance`；不再使用 `xt_assets.cash` 代替可用保证金。
- 模块当前已具备：
  - 独立 xtquant 查询 client 与 worker 骨架
  - `pm_credit_asset_snapshots` / `pm_current_state` / `pm_strategy_decisions` 仓库层
  - 三态策略：`ALLOW_OPEN` / `HOLDING_ONLY` / `FORCE_PROFIT_REDUCE`
  - 状态过旧保护与默认兜底 `HOLDING_ONLY`
  - `OrderSubmitService` 中仅对 `source=strategy` 生效的仓位门禁
- Guardian 目前只消费 `FORCE_PROFIT_REDUCE` 下的占位标志：当盈利卖点卖出被允许时，会在队列 payload 中透传 `position_management_force_profit_reduce=true` 与 `position_management_profit_reduce_mode=guardian_placeholder`，后续真正的强制卖出算法仍待专门 RFC/任务细化。

## 8. 建议的下一步（仅建议，不直接落地）

若要正式进入迁移与重构，建议先补齐“治理资料”再动代码：

1) **建立首批 RFC 清单**：按子系统拆分（配置/数据采集/信号/实盘/工作流/Web UI/网关），并在 `docs/migration/progress.md` 登记。
2) **明确边界与依赖**：每个 RFC 指定负责范围、禁止依赖、对外入口（CLI/API/worker）与验收方式。
3) **先从公共入口收敛**：优先处理 CLI/API/config 这种“接口面”一致性问题（因为 CI 已将其纳入 breaking-change 门禁）。
4) **补齐最小可用测试**：至少为关键数据结构与参数读取（dynaconf + params）补上单测，逐步让 CI 可“阻止回归”。
